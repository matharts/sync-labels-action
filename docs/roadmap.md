# MathArts Sync Labels 版本路线图

更新日期：2026-07-15

本路线图供维护者确定版本边界和发布顺序。项目先发布 `main` 已完成的安全能力，再扩展范围控制、故障诊断和计划审计。性能优化必须由运行数据触发。每个次版本只交付一个主要结果，前一版本完成生产验证后才启动下一版本。

## 版本概览

版本按风险和依赖顺序排列。目标窗口用于排序，不代替发布验收：

| 版本     | 目标窗口 | 状态     | 主要结果                           |
| -------- | -------- | -------- | ---------------------------------- |
| `v1.4.0` | 2026-07  | 已发布   | 发布 Node.js 24 与整次运行删除保护 |
| `v1.5.0` | 2026-08  | 已规划   | 控制仓库范围并支持离线配置校验     |
| `v1.6.0` | 2026-Q3  | 条件规划 | 提供稳定、可自动处理的故障诊断     |
| `v1.7.0` | 2026-Q4  | 条件规划 | 输出稳定、可归档的运行计划         |
| `v1.8.0` | 待定     | 条件规划 | 根据实测数据降低多仓库同步耗时     |

## 当前发布与开发基线

`v1.4.0` 是当前公开版本，并保留以下 v1.3 兼容性契约：

- 默认 dry-run，并区分组织受管标签和仓库扩展标签
- 规划创建、更新、重命名、删除、未变化和保留操作
- 读取请求可以重试，写入请求不重试
- 单仓库失败不阻止后续仓库，并保留部分完成计数
- GitHub Actions 任务摘要和结构化 Action 输出使用同一运行结果

本版本新增以下能力：

- 原生 Node.js 24 Action、TypeScript 严格模式、pnpm 锁文件和提交的 `dist/index.js`
- 配置、仓库选择、GitHub 访问、规划和执行之间的模块边界
- 在首个写请求前规划全部仓库的不可变 `RunPlan`
- 禁止删除、单仓库删除上限和整次运行删除上限
- 对照 Ruby v1.3 行为快照的迁移等价性测试

`v1.4.0` 已完成组织级预览、单仓库真实写入和零差异复查，并通过本地检查、可复现 bundle 比对和 `main` 持续集成（CI）。[正式版本](https://github.com/matharts/sync-labels-action/releases/tag/v1.4.0)的 Action 运行时代码与经过生产演练的候选版本一致；发布后的 `main` 仅增加版本引用和路线图更新，Action 运行时代码与正式版本一致。

## v1.4.0：发布 Node.js 24 与整次运行安全

本版本把 `main` 收敛为可发布、可回滚且经过真实写入验证的版本。除安全报告和发布材料外，不增加策略能力。

### v1.4.0 交付范围

本版本完成以下工作：

1. 让任务摘要区分 `planning`、`safety` 和 `execution` 阶段
2. 在安全阻止时展示计划删除量和触发的限制
3. 为 MathArts 生产策略配置显式删除规则和保守阈值
4. 使用候选提交完成全组织 dry-run、单仓库真实写入和零差异复查
5. 发布候选版本 `v1.4.0-rc.1`，观察一次计划任务和一次写入演练
6. 同步 `package.json`、README 固定提交哈希、发布说明和版本标签

`repositories.exclude`、`validate_only`、类型化错误、计划文件和审批绑定不属于本版本。

### v1.4.0 验收条件

发布验收已确认：

- 删除规则或任一删除上限阻止运行时，写请求数为零
- 所有仓库在首个写请求前完成规划
- 单仓库规划或执行失败不阻止后续安全计划，且部分完成计数准确
- 未配置 `safety` 的 v1.3 合法配置保持原有计划和最终状态
- Node.js 24 单仓库写入成功，随后 dry-run 的 `changed` 为 `false`
- 任务摘要展示失败阶段、计划删除量和安全阻止原因

## v1.5.0：控制仓库范围并离线校验配置

本版本让维护者在不扩大同步范围、不访问 GitHub 应用程序编程接口（API）的情况下校验配置。

### v1.5.0 交付范围

本版本完成以下工作：

- 为 `label-policy.yml` 的 `repositories` 增加 `exclude`
- 固定选择顺序：全部仓库或 `include` → `exclude` → 可选 `repository` input
- 拒绝同时出现在 `include` 和 `exclude` 中的仓库
- 显式选择被排除仓库时返回错误，不静默跳过
- 增加 `validate_only` input，只解析并交叉校验标签和策略文件
- 提供复用 `GovernanceConfig` 的本地校验命令

### v1.5.0 验收条件

满足以下条件后发布：

- 全部仓库、仅 `include`、仅 `exclude`、组合规则和显式单仓库都有接口测试
- archived、disabled、fork 和 `exclude` 的组合不会扩大同步范围
- `validate_only` 不要求 token 或 owner，在断网环境中的 API 调用数为零
- 本地命令与 Action 对同一配置返回相同的校验结果
- 未设置新 input 的 v1.4 workflow 行为不变

## v1.6.0：稳定故障诊断契约

本版本让维护者和自动化稳定判断失败阶段与处理方式。项目只在 `v1.5.0` 发布后收集到真实配置、权限或网络错误样本时启动该版本。

### v1.6.0 候选范围

根据真实错误样本确定最终范围。诊断契约区分失败阶段和可处理原因：

- 稳定失败阶段：`configuration`、`repository_selection`、`planning`、`safety` 和 `execution`
- 最低原因类别：`authentication`、`authorization`、`rate_limit`、`network`、`invalid_response`、`policy` 和 `unknown`
- 由 `GitHubClient` 统一把响应状态、响应头和网络错误映射为原因类别
- 在运行结果中保留失败阶段和经过筛选的原因类别
- 增加 `failed_repositories` output，返回按仓库全名排序的 JSON（JavaScript Object Notation）数组
- 在任务摘要中按阶段和类别提供处置建议

### v1.6.0 验收条件

满足以下条件后发布：

- 每个失败阶段和原因类别都有覆盖 Action 入口的契约测试
- 权限不足、限流、服务端错误和暂时性网络失败不会混淆
- `failed_repositories` 在零失败、单失败和多失败时格式合法、顺序稳定
- 读取重试边界与 v1.5 兼容，写请求仍不自动重试
- 日志、任务摘要和 Action 输出通过敏感字段回归测试

没有真实错误样本时，项目继续维护 `v1.5.x`，不启动 `v1.6.0`。

## v1.7.0：输出可归档运行计划

本版本为长期留档、变更比较或审批提供稳定计划。至少一个使用方需要这些能力，并能证明任务摘要不足时，项目才启动该版本。

### v1.7.0 候选范围

本版本候选能力包括：

- 输出 `schema_version: 1` 的规范化 JSON 计划
- 记录目标仓库、操作、删除原因、汇总计数和规划失败
- 增加 `plan_path` 和 `plan_digest` outputs
- 使用规范化计划内容计算 SHA-256 摘要
- 按仓库全名和固定操作顺序序列化计划
- 提供上传 artifact 和比较计划的 README 示例

时间戳、临时路径和日志文本不参与摘要。

### v1.7.0 验收条件

满足以下条件后发布：

- API 返回顺序变化不改变计划 JSON 或摘要
- 配置、策略、仓库集合或现有标签变化会改变摘要
- JSON 格式有逐字段契约测试和兼容性规则
- 计划不包含 token、Authorization header、未筛选响应正文或 runner 敏感路径

`expected_plan_digest` 不属于 `v1.7.0`。计划格式经过至少一个版本的实际归档后，再根据审批需求决定是否增加写入门禁。

## v1.8.0：根据实测优化多仓库性能

本版本只处理经过测量的性能瓶颈。项目必须先记录至少两次代表性全组织运行，覆盖仓库数、请求数、阶段耗时、重试和限流。数据必须证明串行网络请求是主要瓶颈，实施 Issue 还必须在开发前定义改善目标。

### v1.8.0 候选范围

项目按以下顺序评估优化：

1. 建立可复现基线，分别记录网络等待、本地计算和重试等待
2. 优先为只读规划增加保序、有上限的并发
3. 使用 GitHub 分页响应的下一页信息，避免整页结果触发额外空页请求
4. 测量证明仓库选择影响目标时，再优化对应读取路径
5. 只读规划仍无法达到目标时，再评估跨仓库写入并发

同一仓库内的写入继续串行。只有微基准证明本地规划影响目标时，项目才优化 `SyncPlanner`。项目不引入远程标签缓存或受管前缀树，除非后续数据证明收益高于一致性和复杂度成本。

### v1.8.0 验收条件

满足以下条件后发布：

- 实施 Issue 在开发前固定工作负载、改善目标和测量方法
- 代表性工作负载达到目标，并公开基线和结果
- 并发有明确上限，测试覆盖在途请求数、乱序完成、重试、限流和规划失败
- 所有仓库仍在首个写请求前完成规划和整次运行安全检查
- 仓库结果、部分失败计数、任务摘要、Action 输出和日志顺序保持确定
- 同一仓库的写入顺序、首错停止、完成计数和写入不重试语义不变
- 请求数和 API 失败率不高于基线；未达到目标时不发布

## 发布与维护规则

路线图只维护版本边界和顺序。GitHub Milestone 管理版本，GitHub Issues 保存产品需求、实现任务和验收证据：

- 每个实现 Issue 由一个聚焦的 Pull Request 完成
- 发布演练、文档同步和正式发布分别跟踪
- 每个次版本先发布候选版本，再完成对应的运行演练
- 补丁版本只处理回归、安全修复和 GitHub API 兼容问题
- 补丁版本不改变同步范围、删除语义、配置格式或 output 含义
- 新版本发布后，前一个次版本继续接收严重回归和安全修复，直到下一个次版本发布
- 只有默认删除行为、配置主版本或既有 output 语义必须改变时，才规划 `v2.0.0`

## 共同发布门槛

每个版本都必须满足以下条件：

- Node.js 24 类型检查、行为测试、可复现 bundle 和 workflow lint 全部通过
- 第三方 Action 固定到完整提交哈希或容器 digest
- 配置、仓库选择、规划、执行、报告和主入口都有接口级行为测试
- README、Action metadata、路线图、版本元数据和发布说明一致
- 写入行为发生变化时，完成 dry-run、单仓库写入和失败恢复演练
- 日志、计划、任务摘要和 Action 输出不暴露令牌或敏感响应字段

## 暂不规划的能力

以下能力不进入当前路线图：

- 同步 Milestone、Project 字段、Issue 内容或负责人
- 自动创建或分发 GitHub App 凭据
- 自动重试结果未知的创建、更新、重命名或删除请求
- 未满足 `v1.8.0` 启动条件的并发、缓存或批处理优化
- 只有一个使用方需要的新策略字段
